# AI駆動開発時代の見積もり・採算管理・アップセル基盤

## 1. コンセプト

本プロダクトは、一般的なプロジェクト管理ツールではない。

> 顧客価値で売り、標準価格で受注し、自社の開発力で利益を出すための、システム開発向け見積もり・採算管理基盤

受注前には、顧客の課題から標準的な工数・価格・非機能要件・期待効果を算出する。  
受注後は、GitHubなどの実働データと連携し、標準工数に対して自社がどれだけ高効率に開発できたかを計測する。

さらに、開発中に発生した追加要望を変更管理表へ蓄積し、次期バージョンの投資案件として提案する。

---

## 2. 基本となる事業モデル

プロジェクトで利益を出す方法には、大きく二つある。

### 2.1 スペシャリスト型

高い能力を持つ人材を投入し、その原価を上回る価値を生み出す。

- 意思決定の高速化
- 手戻りの削減
- 障害や失敗の回避
- リリースの前倒し
- 売上向上
- 将来の運用コスト削減

### 2.2 標準化・効率化型

市場標準価格で受注し、自動化・再利用・AI活用・工程改善によって実行原価を下げる。

```text
案件利益 = 受注価格 - 実行原価
```

本プロダクトでは、後者を中心に据える。

> 一般的な開発会社なら何人月かかる仕事を、自社はいくらの原価で完成させられたか

を定量化し、その差分を利益として捉える。

## 3. 顧客への訴求

顧客に訴求するのは、自社のAI活用や工数削減ではない。

顧客に提示するのは、システム導入によって顧客自身が得られる経済効果である。

### 主な削減対象

- 手作業の人件費
- 二重入力や転記
- 入力ミスと再作業
- 確認待ち
- 外注費
- 既存システム利用料
- 属人化による教育・引継ぎコスト
- 売上機会の損失
- 障害や不正による損失

### 提案例

```text
現在の年間業務コスト      3,000万円
導入後の年間業務コスト    1,200万円
年間削減効果              1,800万円

標準開発価格              1,200万〜1,500万円
投資回収期間              8〜10か月
```

自社の実行原価は、顧客への価格説明の中心には置かない。

## 4. 標準価格の算出

標準価格は、自社の見積もりではなく、一般的な開発会社が標準的な体制と工程で開発した場合の価格として算出する。

### 主なデータソース

- IPAのソフトウェア開発データ
- IPAの非機能要求グレード
- IPAのCoBRA法などの工数変動要因
- 公開されている技術者単価
- 業界団体や官公庁の統計
- 公開された開発費事例
- LLMが持つ一般的な知識
- 自社や導入企業の匿名実績データ

LLMには価格を直接決めさせるのではなく、次の役割を持たせる。

- 提案書やヒアリングメモから機能を抽出
- 画面、API、バッチ、外部連携を分解
- 非機能要件の不足を検出
- システム規模と難易度を分類
- 類似案件を検索
- 工数変動要因を整理
- 見積もり根拠を文章化

### 出力する三つの見積もり

#### 市場標準見積もり

```text
標準工数    18人月
標準価格    2,000万〜2,500万円
標準期間    6〜9か月
```

#### 自社実行見積もり

```text
自社予測工数    7.5人月
自社予測原価    850万円
予測期間        3〜4か月
```

#### 推奨受注価格

```text
顧客価値        年間3,000万円
市場標準価格    2,000万〜2,500万円
自社予測原価    850万円
推奨受注価格    1,800万〜2,200万円
```

受注価格は、自社原価の積み上げではなく、顧客価値と市場標準価格から決める。

## 5. 非機能要件の自動補完

顧客は機能要件を話しても、非機能要件には言及しないことが多い。

そのため、明示されていない場合は、システム規模・業種・利用者・データ特性に応じて標準的な非機能要件を自動適用する。

### 主な対象

- 可用性
- 性能
- 拡張性
- バックアップ
- RPO・RTO
- 監視
- ログ
- 監査証跡
- 権限管理
- セキュリティ
- データ暗号化
- 脆弱性対策
- CI/CD
- ステージング環境
- 障害対応
- データ保持
- 法規制・業界規制

### システムグレード例

| グレード | 用途 | 主な特徴 |
|---|---|---|
| Grade 1 | PoC・試作品 | 最低限の監視、日次バックアップ |
| Grade 2 | 社内業務システム | 操作ログ、CI/CD、復旧手順 |
| Grade 3 | 顧客向けサービス | 冗長化、WAF、監査ログ、負荷試験 |
| Grade 4 | 重要業務・規制領域 | 厳格なRPO/RTO、DR、第三者診断 |

非機能要件が見積もりに与えた影響も明示する。

```text
基本機能開発              7.2人月
テスト・品質保証          1.8人月
セキュリティ対応          0.8人月
監視・ログ                0.6人月
CI/CD・環境構築           0.5人月
PM・要件調整              1.3人月
リスクバッファ            1.0人月
合計                     13.2人月
```

## 6. 見積もり結果の表現

見積もりは一点ではなく、レンジと確度で出す。

```text
楽観値      9.5人月
標準値     12.5人月
慎重値     16.0人月
```

さらに、見積もりを押し上げる要因を表示する。

```text
既存システム仕様が不明        +15〜25%
データ移行品質が未確認        +5〜15%
決済・返金フローが複雑        +8〜12%
承認者が未確定                +5〜10%
```

## 7. 実働との連携

受注後は、見積もりをプロジェクトへ変換する。

```text
プロジェクト
└── マイルストーン
    └── 機能・成果物
        └── 実装タスク
            └── GitHub Issue / Pull Request
```

### GitHub連携

- GitHub AppでOrganization・Repositoryを接続
- タスクキーをPRタイトル・本文・ブランチ名から認識
- PR作成で開発中へ更新
- レビュー依頼でレビュー中へ更新
- CI成功でマージ待ちへ更新
- PRマージで実装完了へ更新
- 本番デプロイ成功で提供済みへ更新
- 関連するバグ修正PRを追跡
- 未紐付けPRを検出

### 完了条件

案件ごとにDefinition of Doneを設定する。

```text
PRマージ
ステージング反映
本番反映
PM承認
顧客承認
```

## 8. 進捗の測定

タスク件数による進捗率は使わない。

難しい機能と軽微な修正を同じ1件として扱うと、実態を表せないためである。

### 重み付き進捗

| サイズ | ポイント |
|---|---:|
| XS | 1 |
| S | 2 |
| M | 5 |
| L | 8 |
| XL | 13 |

```text
作業進捗率 =
完了ポイント合計 / 現在スコープのポイント合計
```

### 複数の進捗指標

単一の進捗率ではなく、最低限以下を分ける。

```text
作業進捗率          55%
原価消化率          73%
スコープ増加率      18%
必須機能完了率      60%
マイルストーン      2 / 5
```

### 当初計画と現在スコープ

```text
当初計画        100pt
現在スコープ    125pt
完了             60pt
```

```text
当初計画に対する完了率      60%
現在スコープに対する完了率  48%
スコープ増加率              25%
```

これにより、遅延の原因が作業速度なのか、追加要求なのか、初期見積もりなのかを分けて判断できる。

## 9. 標準見積もりと実績の比較

プロジェクト終了後には、標準工数・自社予測・実績を比較する。

| 機能 | 標準工数 | 自社予測 | 実績 | 標準比削減率 | 品質 |
|---|---:|---:|---:|---:|---|
| 認証・権限 | 180h | 100h | 82h | 54% | 良好 |
| 商品管理 | 320h | 180h | 195h | 39% | 良好 |
| 決済 | 240h | 160h | 145h | 40% | 要注意 |
| 管理画面 | 300h | 130h | 105h | 65% | 良好 |

```text
標準工数        2,400時間
自社予測        1,350時間
実績            1,180時間

標準比生産性    2.03倍
受注額          2,400万円
実績原価          980万円
粗利益          1,420万円
粗利益率          59.2%
```

## 10. 品質を含めた生産性

時短だけでは生産性とは呼ばない。

```text
生産性 =
標準工数に対する削減率
× 品質係数
× スコープ達成率
```

### 品質指標

- 本番障害件数
- リリース後の修正工数
- バグ再発率
- PR差し戻し率
- CI失敗率
- テストカバレッジ
- セキュリティ指摘
- 顧客受入れ不合格率
- 運用負荷
- 保守性

```text

```

これにより、新規案件に対して自社固有の実行予測を出せる。

## 12. 変更管理とアップセル

開発中に発生した要望は、その場で無償対応せず、変更管理表へ登録する。

変更管理は原価防衛だけでなく、次期バージョンの営業パイプラインとして扱う。

### 要望の分類

| 区分 | 意味 | 扱い |
|---|---|---|
| 不具合 | 合意済み仕様を満たしていない | 原則契約内 |
| 仕様漏れ | 双方の見落とし | 経緯を見て判断 |
| 仕様変更 | 合意済み内容を変更 | 追加見積もり |
| 機能拡張 | 新しい価値を生む機能 | 次期アップセル |

### 変更管理表

| 項目 | 内容 |
|---|---|
| 要望 | 顧客が希望していること |
| 発生背景 | なぜ必要になったか |
| 現在の損失 | 時間、外注費、ミス、機会損失 |
| 期待効果 | 導入後に改善すること |
| 標準工数 | 一般的に必要な工数 |
| 自社予測工数 | 自社なら必要な工数 |
| 推奨価格 | 顧客価値と標準価格から算出 |
| 優先度 | 効果、緊急度、依存関係 |
| 対応バージョン | v1.1、v2.0など |
| 状態 | 発見、評価、提案、承認、保留 |

### アップセル例

```text
要望
受注データのCSV出力

現状
担当者2名が月20時間を手作業で集計

年間削減効果
20時間 × 12か月 × 4,000円 = 96万円

標準開発価格
80万〜120万円

自社予測原価
25万円

提案価格
80万円
```

### バージョン単位での提案

```text
Version 1.1：運用負荷削減
- CSV出力
- 一括更新
- 通知設定

提案価格        180万円
年間削減効果    320万円
```

開発中に得られる業務理解を、次の投資提案につなげる。

## 13. プロダクト全体のループ

```text
顧客の課題を入力
    ↓
顧客の経済効果を算出
    ↓
標準工数・標準価格を算出
    ↓
自社実行工数・原価を予測
    ↓
顧客価値と市場価格を基準に受注
    ↓
GitHub連携で実働を取得
    ↓
標準工数との差分を利益として計測
    ↓
品質・手戻りを含めて生産性を評価
    ↓
実績を見積もり・教育・開発基盤へ反映
    ↓
開発中の追加要望を変更管理表へ蓄積
    ↓
次期バージョンとしてアップセル
```

## 14. 自社のMOAT

AIモデルや開発ツールそのものは、長期的なMOATになりにくい。

自社の競争優位は、以下の積で作る。

```text
優秀な人材
× 人材教育
× 開発ハーネス
× 標準化
× 再利用資産
× 実績データ
```

### 人材教育

- 要件を構造化する能力
- 非機能要件を判断する能力
- アーキテクチャ設計
- AI生成物のレビュー
- 品質と速度の両立
- フロント・バックエンド・インフラの横断理解
- 顧客の事業価値を理解する能力

### 仕組み化

- AI開発ハーネス
- コーディング規約
- 設計テンプレート
- テスト生成基盤
- CI/CD
- PRレビュー基準
- 認証・決済・管理画面などの再利用部品
- 非機能要件テンプレート
- 見積もりと実績の比較データ
- 障害・手戻りのフィードバック

スペシャリスト個人の能力を、他の人材でも再現できる形に変換する。

## 16. プロダクトのポジショニング

本プロダクトは、単なるプロジェクト管理ツールではない。

> 顧客の課題から経済効果と標準開発価格を算出し、受注後はGitHubの実績と連動して、自社固有の生産性・利益・次期アップセル機会を可視化する開発経営プラットフォーム

### 一言で表すなら

> 標準ならいくらかかる仕事を、自社はいくらで作れたか。

### 事業モデルを表すなら

> 顧客価値で売り、標準価格で受注し、自社の開発力で利益を出す。

### 長期的な競争優位を表すなら

> 標準工数を下回りながら品質を維持できる、人材・仕組み・実績データ。
