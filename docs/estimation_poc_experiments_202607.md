# 見積もりPoC 実験結果とネクストアクション(2026-07-12〜13)

PoC(`docs/estimate_core.md` の設計 v0.1)を実装し、架空要件2件で claude -p による見積もり生成を実打ちした結果のまとめ。

## 実験条件

- 実行: `claude -p --output-format json`(ツール無効)、プロンプトは `prompts/task_breakdown.md`
- 単価: 50,000円/人日(`Project.daily_rate` の既定値。人月100万円相当)
- 見積もり基準: 実務3年程度のエンジニアが従来型開発(AI支援なし)で行う場合の標準工数
- サンプル要件: `docs/samples/` 配下(いずれも架空企業)

## 実験1: 不動産営業管理システム(制約なし)

要件: `docs/samples/real_estate_sales_requirement.md`(売買仲介50名・3店舗。反響取込、追客、物件・媒介契約、商談パイプライン、内見予約、日報、KPIダッシュボード等)

### 1a. references 注入前

- **36タスク・156人日・780万円**
- カテゴリ分類・共通タスク(環境構築、テスト、PM)・リスク要因の指摘はいずれも妥当
- 不明点の指摘が実務的(反響メールの受信方式、ポータル3社のメールサンプル入手が必須、移行データの実態)

### 1b. references 注入後(IPA基準値)

`references/IPA標準見積もり基準値.md` をプロンプトに注入した同一要件での再実行。

- **39タスク・約214人日**(要件定義・要求整理などの上流タスクが追加され、より網羅的・保守的に)
- FP概算(350〜450FP)→ IPA FP生産性中央値でのクロスチェック → 白書2018-2019の少人数チーム補正(5人未満は約1.5〜1.9倍)まで適用し、「5工程相当172人日はレンジ160〜220人日の下限寄り=楽観側なのでバッファ推奨」と**自己検算**した
- **所見**: 基準値の注入で見積もりが上振れした(156→214人日)。「アンカーがないと楽観に寄る」仮説と整合。対外説明の根拠が付く効果も大きい

### 体制感の目安

3人体制なら3ヶ月弱+リスクバッファで **3〜5ヶ月**。標準単価90〜120万円/人月なら **約780万〜1,030万円** が売値レンジ。

## 実験2: 採用管理システム(予算上限500万円)

要件: `docs/samples/ats_startup_requirement.md`(SaaSベンチャー35名。候補者カンバン+滞留アラート、技術課題ワークフロー、Slack通知、ブラインド評価、ファネルCSV。**予算上限500万円を要件に明記**)

### 結果

- 全量積み上げ: **28タスク・106人日 = 530万円** → 予算超過を自ら認識
- スコープ調整提案(初期リリースから外す):
  1. **Googleカレンダー連携(7人日)** — 要件文の「難しければ記録だけでもいい」を根拠に引用
  2. **不採用者データ自動削除バッチ(3人日)** — 法務確認完了まで仕様が固まらないため手動運用に
- 調整後 **96人日 = 480万円(税別)で予算内**。「バッファ4%しかない」と明示し、さらに削る場合の第3案(データ移行を顧客側作業に)まで提示
- IPA基準値でのクロスチェック(110〜130FP ≒ 90〜110人日)と積み上げ106人日の整合を確認
- 「予算固定のため、上振れ時は機能削減で吸収する契約設計を推奨」と**契約設計まで言及**

### 発見されたプロダクト課題

**スコープ調整の提案が文章にしか存在せず、データに反映されない。**
タスクテーブルには外す提案をした2件も含む全28件(530万円)が保存され、UI上の合計は予算超過のまま。Task モデルに「初期スコープ / 次期フェーズ」の区分がないことが原因。

これは構想(`docs/ai_development_estimation_profit_platform.md` §12)の**変更管理表・バージョン単位アップセル**に直結する: 「予算で外したタスク」= 次期フェーズの提案リストとしてデータで持てれば、営業パイプラインになる。

## 全体の評価

| 観点 | 評価 |
|---|---|
| 機能・タスクの網羅性 | ○ 明記されていない必然タスク(認証、マスタ、通知基盤)を補完し、補完した旨を description に明記 |
| 人日の妥当性 | ○ 体感と整合。references 注入でアンカー付きになり対外説明可能に |
| 不明点・リスク指摘 | ◎ 実務で本当に聞くべき質問が出る(メールサンプル、法務要件、権限の厳密範囲) |
| 予算制約への応答 | ◎ 超過認識 → 根拠付きスコープ提案 → バッファまで正直に申告 |
| 対話継続(--resume) | ○ セッション維持・タスク洗い替え・ユーザー編集の保持を確認 |
| データモデルの表現力 | △ スコープ区分・フェーズの概念がなく、調整提案を保存できない |

## ネクストアクション

### プロダクト改修(優先度順)

1. **Task にスコープ区分を追加**(例: `phase` = initial / next / out_of_scope)
   - LLMのJSON出力にも区分を含めさせ、スコープ調整提案をデータ化する
   - 合計・予算判定は「初期スコープのみ」と「全量」を並記
   - 外したタスクは次期フェーズ提案リスト(アップセルパイプラインの原型)として表示
2. **予算・利益率ガードのUI**: プロジェクトに予算上限・想定売価を持たせ、超過/粗利率40%割れの警告を常時表示(議事録 §2.7)
3. **人日編集時の価格追従**: 人日だけ手動編集した場合に価格を単価×人日で再計算するか要判断(現状は追従しない)
4. **estimate.md の組み込み**: 工程掛け率(設計15〜20%、テスト30〜40%、PM10〜15%、予備費10〜20%)レビューを対話のワンアクションにする

### 検証・データ整備

5. **直近実案件1〜2件での検算**(議事録のネクストアクション): 実際の要件を投入し、受注額・実工数との乖離を測る。PoCの精度評価はこれが本丸
6. **references の拡充**: カテゴリ別標準工数テーブル(ck-knowledge の叩き台を検算後に配置)、技術者単価相場を追加
7. **references 注入の前後比較の継続**: 実験1で見積もりが約1.4倍に上振れした。どちらが実態に近いかは 5. の検算で判定する

### 運用メモ

- 生成された見積もりの1件目(実験1)はDB project=2、実験2は project=3 として残置(UIでの動作確認用)
- 既知の制約: 対話1往復に数分かかる(claude -p 同期実行)。UX上はポーリング表示で許容範囲だが、体感短縮は将来課題
